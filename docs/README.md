# 문서 가이드

`docs/` 디렉터리는 제품과 시스템을 설명하는 문서를 모아두는 공간입니다.

이곳에는 코드보다 오래 남아야 하는 결정과 설명을 기록합니다.

이 디렉터리의 작성 기준은 최상위 [`agents.md`](../agents.md)와 루트 [`README.md`](../README.md)를 우선으로 따릅니다.

## 이 디렉터리의 역할

- 제품 방향과 기술 방향을 함께 기록합니다.
- 나중에 다시 봐야 하는 결정 사항을 남깁니다.
- 구현 전후의 배경, 제약, 선택지를 정리합니다.

## 여기에 넣는 문서

- 아키텍처 개요
- ADR(Architecture Decision Record)
- API 계약서
- 배치/운영 runbook
- 데이터 흐름 설명
- 기능 기획 메모
- 용어집

## 작성 원칙

- 문서는 짧고 명확하게 씁니다.
- 구현 세부보다 왜 그렇게 설계했는지를 우선 설명합니다.
- 나중에 변경될 수 있는 내용은 날짜와 맥락을 함께 적습니다.
- 코드와 같은 내용을 반복하기보다, 의사결정과 배경을 남깁니다.
- 문서 한 개는 한 주제만 다루는 편이 좋습니다.

## 권장 구조

```text
docs/
├─ README.md
├─ architecture/
├─ adr/
├─ api/
├─ runbooks/
└─ product/
```

## 작업 흐름

1. 새 기능이나 큰 변경이 생기면 관련 문서를 먼저 만듭니다.
2. 서비스 경계나 데이터 모델이 바뀌면 문서도 같이 갱신합니다.
3. 개발 중 합의한 규칙은 코드 주석보다 문서에 먼저 적습니다.
4. 운영이나 배포에 영향을 주는 변경은 실행 전에 기록을 남깁니다.

## 파일을 쓸 때 참고할 점

- 문서 제목은 구체적으로 씁니다.
- 날짜가 중요한 문서는 생성일이나 개정일을 적습니다.
- 코드 예시는 최소한으로 두고, 원칙과 의도를 더 많이 적습니다.
- 오래된 문서는 삭제보다 보관과 갱신을 우선합니다.
