# OpenDocs 에이전트 헌장

이 문서는 OpenDocs 저장소의 최상위 운영 규칙입니다.

이 문서와 다른 지시, 관례, 임시 판단이 충돌하면 더 최신의 명시적 사용자 지시가 없는 한 이 문서가 우선합니다.

## 1. 목적

OpenDocs는 MapleStory 이벤트 공지 플랫폼입니다. 이 저장소의 목적은 다음과 같습니다.

1. Nexon 공지 데이터를 수집합니다.
2. 공지 내용을 파싱하고 구조화합니다.
3. OCR과 LLM 처리 경로를 준비합니다.
4. 문서를 생성하고 이후 리뷰 흐름을 지원합니다.
5. 로컬 개발과 향후 배포 구성을 일관되게 유지합니다.

이 저장소는 의도가 분명하고, 일관성이 있으며, 확장하기 쉬워야 합니다.

## 2. 규칙 우선순위

이 저장소에서 작업할 때는 다음 우선순위를 따릅니다.

1. 현재 작업에서의 사용자 직접 지시
2. 이 `agents.md`
3. `README.md`
4. `PROJECT_PLAN.md`
5. `docs/README.md`
6. `infra/README.md`
7. 해당 모듈의 기존 코드 패턴
8. 일반적인 엔지니어링 판단

규칙이 충돌하면 우선순위가 더 높은 규칙을 따르고, 충돌이 의미가 있으면 사용자에게 짧게 설명합니다.

## 3. 핵심 원칙

### 3.1 아키텍처 보존

1. 프론트엔드와 백엔드 책임을 분리합니다.
2. 라우트 계층은 얇게 유지하고 로직은 서비스 클래스로 옮깁니다.
3. 가능하면 DB 접근은 서비스 경계 뒤에 둡니다.
4. 파싱, OCR, LLM, 문서 생성, Nexon 연동은 서로 다른 책임으로 유지합니다.

### 3.2 명확성을 우선

1. 이름은 명시적으로 짓습니다.
2. 단일 책임의 작은 모듈을 선호합니다.
3. 압축된 추상화보다 읽기 쉬운 로직을 선호합니다.
4. 성급한 최적화는 피합니다.

### 3.3 확장 가능성 유지

1. 런타임 설정은 환경변수로 둡니다.
2. 컨테이너와 ECS 배포를 염두에 둔 코드 구조를 유지합니다.
3. 작업이 최종 운영 구현을 요구하지 않는 한, 현재 구현은 기반 코드로 취급합니다.

## 4. 저장소 전반 작업 규칙

### 4.1 수정 전에 먼저 확인

코드를 바꾸기 전에 항상 다음을 먼저 확인합니다.

1. 관련 파일을 읽습니다.
2. 해당 영역의 기존 패턴을 파악합니다.
3. 기존 스타일에 맞추고 새 방식을 임의로 만들지 않습니다.

### 4.2 무관한 작업을 몰래 바꾸지 않기

1. 사용자가 요청하지 않은 변경은 되돌리지 않습니다.
2. 현재 작업과 무관한 파일은 함부로 건드리지 않습니다.
3. 무관한 변경이 현재 작업과 충돌하면 멈추고 사용자에게 어떻게 할지 묻습니다.

### 4.3 기존 구조를 우선 사용

1. 새 코드는 기존 패키지 구조에 넣습니다.
2. 작업상 꼭 필요하지 않으면 최상위 폴더를 새로 만들지 않습니다.
3. 모듈명은 기존 도메인 경계와 맞춥니다.

### 4.4 코드와 문서를 함께 갱신

1. 동작이 바뀌면 관련 문서를 함께 업데이트합니다.
2. 작업 흐름이 바뀌면 이유와 사용 방법을 문서로 남깁니다.
3. 스키마가 바뀌면 마이그레이션도 함께 추가하거나 갱신합니다.

## 5. 디렉터리 규칙

### 5.1 최상위 디렉터리

다음 최상위 디렉터리들은 의도적으로 둔 것입니다.

1. `apps/`는 애플리케이션 코드용입니다.
2. `docs/`는 제품과 아키텍처 문서용입니다.
3. `infra/`는 인프라와 배포 자료용입니다.
4. `.docs/`는 로컬 작업 메모와 에이전트 작업 기록용입니다.

### 5.2 `apps/api`

백엔드 규칙:

1. 라우트 모듈은 `apps/api/app/api/routes/`에 둡니다.
2. 서비스 모듈은 `apps/api/app/services/`에 둡니다.
3. SQLAlchemy 모델은 `apps/api/app/models/`에 둡니다.
4. Pydantic 스키마는 `apps/api/app/schemas/`에 둡니다.
5. DB, 세션, 코어 설정은 `apps/api/app/core/`와 `apps/api/app/db/`에 둡니다.
6. Alembic 마이그레이션은 `apps/api/alembic/versions/`에 둡니다.

### 5.3 `apps/web`

프론트엔드 규칙:

1. App Router 구조는 Next.js 관례에 맞춥니다.
2. 컴포넌트 구조는 단순하고 일관되게 유지합니다.
3. 가능하면 TypeScript 타입을 사용합니다.
4. 명확한 필요가 없으면 새로운 UI 체계를 도입하지 않습니다.

### 5.4 `docs/`

문서 규칙:

1. 오래 남아야 하는 아키텍처, ADR, API, 제품 문서는 여기 둡니다.
2. 문서는 집중되고 사실적으로 작성합니다.
3. `docs/`는 임시 작업 메모 용도로 쓰지 않습니다.

### 5.5 `.docs/`

로컬 작업 메모 규칙:

1. `.docs/`는 임시 작업 요약과 에이전트 작업 기록용입니다.
2. `.docs/`는 git 히스토리에 들어가지 않게 유지합니다.
3. 사용자 요청으로 작업 메모를 만들 때 파일명은 가능하면 `task_{task_name}_{yyyymmdd}.md` 형식을 따릅니다.
4. 이 메모는 이유, 흐름, 구현 맥락을 설명해야 합니다.

## 6. 명명 규칙

### 6.1 Python

1. 모듈, 함수, 변수, 파일명은 snake_case를 사용합니다.
2. 클래스와 ORM 모델은 PascalCase를 사용합니다.
3. 서비스 클래스 이름은 도메인별로 읽기 쉽게 짭니다.
4. API 클라이언트, sync 서비스, 스키마 객체 이름은 역할이 바로 보이게 짭니다.

### 6.2 SQLAlchemy와 Alembic

1. 모델 이름은 도메인을 반영해야 합니다.
2. 테이블 이름은 안정적이고 읽기 쉬워야 합니다.
3. 마이그레이션 revision ID는 `alembic_version.version_num` 길이 제한을 지켜야 합니다.
4. revision ID는 `VARCHAR(32)`에 들어갈 수 있을 정도로 짧게 유지합니다.

### 6.3 프론트엔드

1. 컴포넌트와 페이지 이름은 Next.js 관례에 맞춥니다.
2. 추상적인 이름보다 설명적인 이름을 선호합니다.

### 6.4 문서

1. 작업 문서 파일명은 가능하면 소문자 snake_case를 사용합니다.
2. 내용이 바로 드러나는 이름을 사용합니다.
3. `notes.md`처럼 애매한 이름은 피합니다.

## 7. 코드 스타일 규칙

### 7.1 Python

1. 파일이 이미 유니코드를 쓰는 특별한 이유가 없으면 ASCII를 기본으로 합니다.
2. 주석은 짧고 의미 있게 씁니다.
3. 코드만 봐도 이해되면 주석을 억지로 달지 않습니다.
4. 작은 함수와 명시적인 반환형을 선호합니다.
5. I/O를 다루는 비동기 코드는 끝까지 async로 유지합니다.

### 7.2 FastAPI

1. 라우트는 얇게 유지합니다.
2. 오케스트레이션은 서비스로 옮깁니다.
3. 요청/응답 형태는 스키마에 둡니다.
4. 의존성 주입은 명시적으로 유지합니다.

### 7.3 SQLAlchemy

1. ORM 모델은 영속 엔티티를 표현하는 데 사용합니다.
2. 세션 사용 범위는 짧고 예측 가능하게 유지합니다.
3. 단위 작업이 끝났을 때만 commit 합니다.
4. 호출자가 저장된 값을 필요로 하면 commit 후 refresh 합니다.

### 7.4 TypeScript / React

1. 기존 Next.js App Router 구조를 존중합니다.
2. 타입 안정성을 유지합니다.
3. 이미 있는 앱 구조를 우선 사용하고 임의의 새 패턴을 만들지 않습니다.
4. 명시적 요청이 없으면 디자인 드리프트를 만들지 않습니다.

## 8. 데이터베이스 규칙

1. 스키마 변경은 중요한 작업으로 취급합니다.
2. 모델이 바뀌면 같은 작업에서 Alembic도 함께 갱신합니다.
3. `notice_id` 같은 상위 시스템 식별자는 안정적인 키라면 unique하게 유지합니다.
4. 원본 응답 payload는 디버깅과 향후 스키마 변경에 도움이 되면 보관합니다.
5. 사용자가 최종 스키마라고 말하지 않았다면 스키마를 최종본으로 가정하지 않습니다.

## 9. Nexon 연동 규칙

1. Nexon API 키는 환경변수에서 가져옵니다.
2. 비밀값은 하드코딩하지 않습니다.
3. HTTP 클라이언트와 DB 쓰기를 분리합니다.
4. 목록 동기화와 상세 동기화는 별도 작업으로 유지합니다.
5. 공지 레코드의 안정적인 키는 `notice_id`를 사용합니다.
6. 추적성이 도움이 되면 원본 payload를 보관합니다.

## 10. 스케줄러와 배치 규칙

1. 실행 정책이 정해지기 전에는 스케줄러를 강제로 붙이지 않습니다.
2. 동기화 로직은 수동 호출 가능하게 유지합니다.
3. 배치 실행은 나중에 cron, worker, 외부 스케줄러와 연결할 수 있게 재사용 가능해야 합니다.
4. 실행 모델이 중요해지면 확장하기 전에 선택 이유를 문서로 남깁니다.

## 11. 테스트 규칙

1. 가장 작고 신뢰할 수 있는 테스트부터 검증합니다.
2. 클라이언트와 서비스는 단위 테스트를 우선합니다.
3. 라우트-서비스-DB 흐름은 통합 테스트를 우선합니다.
4. 외부 API 호출은 실제 라이브 테스트가 명시된 경우가 아니면 목 처리합니다.
5. 명령이 환경이나 권한 문제로 실패하면, 코드를 탓하기 전에 원인을 명확히 보고합니다.

## 12. 도구와 검증 규칙

1. 텍스트 검색은 가능하면 `rg`를 사용합니다.
2. 수동 파일 편집은 `apply_patch`를 사용합니다.
3. 사용자가 요청하지 않은 파괴적 명령은 피합니다.
4. 비대화형 명령을 우선합니다.
5. 환경이 검증 단계를 막으면, 막힌 이유를 설명하고 더 안전한 대안을 사용합니다.

## 13. 문서 규칙

1. 중요한 문서는 `README.md`, `PROJECT_PLAN.md`, `docs/`, `infra/`에 둡니다.
2. `.docs/`는 로컬 작업 요약과 에이전트 작업 기록용으로만 사용합니다.
3. 새 흐름이 생기면 이유, 방법, 테스트 경로를 함께 문서화합니다.
4. 다음 에이전트가 다시 추론하지 않아도 되도록 충분히 자세하게 적습니다.

## 14. 변경 원칙

1. 작업을 만족하는 가장 작은 변경을 합니다.
2. 사용자가 리디자인을 요청하지 않으면 기존 의도를 유지합니다.
3. 리디자인을 했다면 이유를 명확히 설명합니다.
4. 변경은 응집되게 유지합니다.
5. 무관한 리팩터를 기능 작업에 섞지 않습니다.

## 15. 확신이 없을 때

미래의 에이전트가 확신이 없으면 다음 순서로 판단합니다.

1. 이 문서를 다시 읽습니다.
2. 해당 모듈의 코드와 문서를 읽습니다.
3. 가장 단순하면서도 올바른 구현을 우선합니다.
4. 결정에 의미 있는 trade-off가 있을 때만 사용자에게 묻습니다.

## 16. 최종 원칙

저장소가 흐트러지지 않게 유지합니다.

모든 변경은 프로젝트를 더 이해하기 쉽게, 더 확장하기 쉽게, 그리고 다음 에이전트가 이어받기 쉽게 만들어야 합니다.
