# OpenDocs 프로젝트 계획

OpenDocs는 MapleStory 이벤트 공지를 자동으로 수집하고, OCR과 구조화 과정을 거쳐 문서로 정리하는 플랫폼입니다.

이 문서는 앞으로 어떤 순서로 작업할지, 각 단계에서 무엇을 완성해야 하는지 정리한 기준 문서입니다.

## 문서 기준

- 전체 작업의 1차 기준은 이 파일입니다.
- 문서 작성 기준은 [`docs/README.md`](./docs/README.md)를 따릅니다.
- 인프라 작업은 [`infra/README.md`](./infra/README.md)를 참고합니다.
- 루트 작업 규칙은 [`README.md`](./README.md)를 참고합니다.
- 구현보다 먼저 경계와 책임을 정리합니다.

## 현재 상태

- 1단계 스캐폴드는 완료했습니다.
- 다음 단계부터는 실제 기능보다는 기반을 더 단단히 다집니다.
- 아직 OCR, LLM, 인증/인가, 커뮤니티 리뷰는 구현하지 않습니다.

## 1단계: 스캐폴드

목표:
- 모노레포 기본 구조를 정리합니다.
- 프론트엔드와 백엔드의 기술 스택을 맞춥니다.
- 로컬 개발 환경을 바로 띄울 수 있게 합니다.

완료 기준:
- `apps/web`에 Next.js App Router + TypeScript + Tailwind CSS 구조가 있습니다.
- `apps/api`에 FastAPI + SQLAlchemy 2.0 + Alembic + Pydantic Settings 구조가 있습니다.
- PostgreSQL과 Docker Compose가 연결되어 있습니다.
- `README.md`, `docs/README.md`, `infra/README.md`, `.env.example`가 정리되어 있습니다.

## 2단계: 플랫폼 기반

목표:
- 데이터베이스와 설정 관리의 기초를 다집니다.
- 배포 전제에 맞는 백엔드 구조를 정리합니다.

작업 항목:
- SQLAlchemy 모델과 Alembic 마이그레이션 체계를 정리합니다.
- 공통 설정과 환경변수 규칙을 확정합니다.
- 로깅과 헬스체크 패턴을 통일합니다.
- ECS 친화적인 Docker 이미지 패턴을 준비합니다.

완료 기준:
- 새 모델 추가 시 마이그레이션을 바로 만들 수 있습니다.
- 로컬과 배포 환경이 동일한 설정 체계를 사용합니다.
- 백엔드가 헬스체크 엔드포인트를 제공합니다.

## 3단계: 수집 및 파싱 경계

목표:
- Nexon 공지 수집과 파싱의 책임 경계를 나눕니다.
- HTML 공지와 이미지 공지를 다룰 수 있는 구조를 만듭니다.

작업 항목:
- Nexon OpenAPI 연동 구조를 정리합니다.
- HTML 파서와 이미지 추출기 경계를 분리합니다.
- 공지 데이터의 원본, 정규화, 추출 결과를 구분합니다.

완료 기준:
- 공지 수집과 파싱이 독립된 서비스로 분리됩니다.
- HTML과 이미지 입력 경로가 서로 섞이지 않습니다.

## 4단계: OCR 및 LLM 파이프라인

목표:
- 이미지 기반 공지에서 구조화 데이터를 만들 수 있는 파이프라인을 준비합니다.

작업 항목:
- OCR 서비스와 프롬프트 경계를 정리합니다.
- LLM 서비스와 프롬프트 경계를 정리합니다.
- JSON 생성과 Markdown 생성 책임을 분리합니다.

완료 기준:
- OCR, JSON 생성, Markdown 생성이 각각 독립된 모듈로 존재합니다.
- 실제 구현은 나중에 추가하더라도 경계가 명확합니다.

## 5단계: 문서 생성

목표:
- 구조화된 공지 데이터를 사람이 읽을 수 있는 문서로 정리합니다.

작업 항목:
- 문서 서비스 레이어를 정리합니다.
- Markdown 출력 규칙을 정의합니다.
- 문서 저장 및 버전 관리 방식을 설계합니다.

완료 기준:
- 문서 생성 책임이 하나의 서비스 경계로 모입니다.
- 출력 포맷과 저장 규칙이 문서화됩니다.

## 6단계: 협업 준비

목표:
- 커뮤니티 리뷰와 향후 협업 기능을 위한 기반을 마련합니다.

작업 항목:
- 리뷰, 승인, 코멘트 흐름을 설계합니다.
- 워크스페이스와 권한 모델을 정리합니다.
- 변경 이력과 감사 로그 기준을 잡습니다.

완료 기준:
- 협업 기능을 붙일 수 있는 데이터와 API 구조가 준비됩니다.
- 인증/인가 설계가 이 단계에서 자연스럽게 이어집니다.

## 7단계: 인프라

목표:
- 실제 배포를 위한 인프라 구성을 준비합니다.

작업 항목:
- Terraform 구조를 `infra/terraform` 아래에 정리합니다.
- ECS task / service 정의를 준비합니다.
- 네트워크, 보안 그룹, 비밀값 관리 방식을 정리합니다.
- 배포 및 롤백 절차를 문서화합니다.

완료 기준:
- 로컬 개발용 구성과 배포용 구성이 분리됩니다.
- ECS로 옮기기 쉬운 컨테이너/환경변수 구조가 유지됩니다.
